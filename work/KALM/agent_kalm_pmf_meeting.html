AI 中心 Agent KALM 會議｜第三點 PMF 討論草案

Agent KALM：從「裝置與模型」收斂到「學習與知識服務」

本文件目的不是替會議做單一結論，而是建立一個可共同討論的框架：
把 To C、To B、Internal 三個 PMF 戰場拆清楚，讓硬體、模型、RAG、MCP、校務系統整合回到正確位置。

To C以整理好的知識主題為入口，提供個人 Learning Agent。
To B以學校管理系統與校務整合為核心，支援導入與驗收。
Internal先在出版社內部企劃／編輯流程驗證知識生產能力。
DeviceAI PC、Phone、Glasses、Personal Device 是載體，不是 PMF 本身。
核心主張 三個戰場 To C To B Internal 技術定位 路線圖 會議討論
先把問題收斂

第三點不應該問
「硬體、模型、Engine、RAG、MCP 哪個才是 PMF」

這些都是解決方案能力，不是市場本身。
真正要討論的是：Agent KALM 要服務哪一種學習行為、哪一種組織場景、哪一種知識工作流程。

建議把 Agent KALM 定義成：
「以整理好的知識主題為基礎，協助個人學習、學校管理與內部知識生產的 Learning / Knowledge Agent。」
不是主軸 硬體不是 PMF

AI PC、手機、眼鏡、平板或其他個人裝置，是 KALM 的運行載體與體驗入口。

不是主軸 模型不是 PMF

模型能力決定上限，但市場購買的是學習成效、管理能力與知識工作效率。

應該是主軸 知識服務才是 PMF

KALM 的差異化來自「已整理知識庫 + 學習流程 + 組織管理 + 系統整合」。

建議討論框架

三個 PMF 戰場：不要混在同一題裡討論

Agent KALM 可以有共同底層，但三個市場的購買理由、使用者、驗收方式完全不同。
會議時建議分開討論，避免 To C 的產品想像、To B 的採購邏輯、Internal 的流程改善互相混淆。

戰場一：To C

To C 讓使用者進入「已整理好的知識主題」

KALM 不應該只是泛用聊天 Agent，而應該像一個有內容、有路線、有練習、有評估的 Learning Service。

To C 的產品入口不建議是：「你想學什麼？」
這會讓產品變成一般 AI 聊天工具，很難跟 ChatGPT、Gemini、Claude、NotebookLM 區隔。
更好的入口是：「這裡有一組已被整理、標註、結構化、可學習與可測驗的知識主題。」

To C 的核心產品定義

KALM 是一個基於整理好知識庫的 Learning Agent，
讓使用者沿著可信任的知識主題，透過解說、練習、情境模擬、測驗、回饋與個人記憶，逐步把知識變成能力。
內容資產 課本內容

高中職教材、章節架構、知識點、題目、解析、常見錯誤、素養題型。

技能學習 商業口說英文

會議、簡報、電話、接待、談判、Email、面試等情境式訓練。

生活決策 買車／買房／保險／理財

知識科普、流程說明、檢查清單、試算工具、風險提醒與決策輔助。


建議的 To C 使用流程

1選擇知識主題從 DB / RAG / MD 中已整理好的主題進入。
2判斷程度與目的用短測驗、問答或情境題判斷使用者位置。
3生成學習路線不是憑空生成，而是基於既有主題結構重新排列。
4練習與回饋題目、口說、案例、任務、情境模擬。
5累積個人學習記憶記錄弱點、偏好、完成度、可推薦下一主題。

To C 產品不只要「問答」，而要有四種模式

學習模式

解釋概念、建立路線、每日任務、進度追蹤。

練習模式

選擇題、申論、口說、案例題、情境題、錯誤回饋。

顧問模式

買車、買房、保險、理財等知識導向建議與檢查清單。

伴學模式

提醒、鼓勵、複習、弱點追蹤、下一步推薦。


To C 可以優先驗證的主題池

主題類型適合原因可做能力驗證指標
高中職課本內容 公司已有教材資產，知識結構明確，能與出版核心能力連結。 章節導讀、知識點說明、測驗、錯題分析、素養題輔導。 完成率、答題進步、章節回訪、錯題修正率。
商業口說英文 社會人士與學生都可使用，場景明確，適合做 AI 互動。 角色扮演、會議對話、簡報問答、面試模擬、即時修正。 每日練習次數、錄音／文字輸出量、連續使用天數。
保險／理財／投資入門 社會人士痛點強，但需注意合規，宜定位為知識教育與試算輔助。 名詞解釋、風險分級、流程說明、試算、檢查清單。 主題完成率、工具使用次數、收藏／分享、付費意願。
買車／買房決策知識 人生重大決策，資訊分散，適合整理成知識路線與工具包。 預算試算、流程說明、比較表、風險提醒、文件清單。 回訪、試算使用、清單完成、方案比較次數。
戰場二：To B

To B 的重點不是「學生端 Agent」，
而是「學校可管理、可整合、可驗收」

學校與公家機關專案買的不是一個聊天機器人，而是一套可以被納入教學與行政流程的 AI 學習管理系統。

To B 版本的 KALM 應該被定位成：「AI Learning Management Layer」，
也就是介於學生學習、教師教學、行政管理、校務資料之間的 AI 學習管理層。
老師／行政視角 學校導入真正會問的問題
  • 學生使用狀況能不能看？
  • 老師能不能派任務、看弱點？
  • 行政能不能匯出成果報告？
  • 能不能對接校務系統、LMS、SSO？
  • 資料格式、個資、權限、稽核怎麼處理？
  • 公部門計畫如何驗收？
不該只做 To B 常見失焦
  • 只做學生聊天功能，沒有管理端。
  • 只展示 AI 回答能力，沒有學習歷程。
  • 只做漂亮 Demo，沒有資料匯入／匯出。
  • 沒有跟校務流程整合，變成額外負擔。
  • 沒有報表與驗收資料，專案很難結案。

To B 系統層級建議

1學生端使用 KALM 學習、練習、問答、完成任務。
2教師端建立任務、查看班級弱點、追蹤學生進度。
3行政端管理帳號、班級、課程、授權與成果報表。
4校務整合層SSO、校務系統、LMS、成績／學籍／課程資料。
5專案驗收層使用數據、學習成效、導入報告、稽核紀錄。

校務系統整合：不要只想 API，也要準備多種資料路徑

整合方式適用情境優點風險／注意事項
API 整合 學校或平台已有穩定系統與技術窗口。 即時性高、權限可控、可雙向同步。 需規格文件、開發排程、測試環境與維運責任。
Raw Data 匯入／匯出 學校資訊系統封閉，短期無法開 API。 導入門檻低，適合試辦或專案初期。 需定義 CSV / Excel / JSON 格式、欄位清洗、匯入頻率與錯誤處理。
資料串流／批次同步 需定期同步大量學習紀錄、任務完成、測驗結果。 可支援規模化、報表與資料倉儲。 需處理資料延遲、重複、失敗重送、權限與個資保護。
SSO / 帳號串接 學生與老師需要用既有校園帳號登入。 降低註冊阻力，便於權限管理。 需要處理角色、班級、離校、轉班、停用帳號等生命週期。

To B 最小可行方案

建議 MVP 不要先做全校級大平台。
先做「一校、一班、一門課、一組知識主題」的閉環：學校名單匯入 → 老師派任務 → 學生使用 KALM → 系統記錄學習歷程 → 老師看 Dashboard → 行政匯出成果報告。這條閉環跑通，比展示一堆 AI 功能更有 PMF 價值。
戰場三：Internal

內部 PMF 是最容易先驗證的起點：
出版社的教材生產與知識管理 Agent

公司內部已有課本、題庫、課綱、章節、教案、教師手冊、舊版教材、圖片資源、編輯規範與數位內容需求。
這些不是普通資料，而是可轉換為 To C 與 To B 能力的核心資產。

企劃 新產品與新教材企劃

競品整理、章節規劃、知識主題拆解、學習路線設計、數位產品腳本。

編輯 教材整理與改版

課綱對照、版本比對、重點摘要、題目分類、錯字格式檢查。

營運／數位 內容轉製與上架

將紙本內容轉成知識卡、題庫、RAG chunk、學習任務與 App 內容。


Internal First 的價值

1先處理真資料不是空泛 Demo，而是用公司實際教材。
2先服務真使用者企劃、編輯、數位團隊每天會遇到這些痛點。
3先打磨流程讓 Agent 進入現有工作流，而不是額外增加負擔。
4累積內容結構教材被整理成未來 To C / To B 可用的知識資產。
5轉外部產品內部能力成熟後，再包成學校與個人可用服務。

建議先選 3 個內部驗證情境

驗證情境Agent 工作內容可量化指標外部轉化價值
課綱對照 Agent 比對教材章節與課綱條目，找缺漏、重複、需要補強之處。 人工比對時間降低、缺漏發現率、編輯採納率。 To B 可變成教師備課與課程檢核工具。
題目標記 Agent 為題目標註章節、知識點、難度、題型、核心能力與常見錯誤。 標記速度、人工修正率、一致性、可搜尋性。 To C 可支援個人練習與弱點分析。
教材轉製 Agent 將紙本章節轉成知識卡、學習任務、測驗、摘要、互動腳本。 轉製時間、可上架比例、內容審核通過率。 To C / To B 可直接形成主題課程與學習模組。
技術如何回到產品主軸

硬體、模型、Engine、RAG、MCP 的正確位置

技術不是不重要，而是要避免反過來主導 PMF。更好的說法是：KALM 先有學習與知識服務場景，再決定哪些技術能力是必要支撐。

建議推進路線

不是三條線同時亂做，而是用同一個底層逐步外擴

最穩的策略不是一開始就宣布 To C、To B、Internal 全部都做，而是先建立共同底層，再選擇最容易驗證的場景往外推。

建議順序
Internal First → To B Second → To C Selective Launch

先用內部教材與流程打磨知識結構、RAG、標註、轉製與 Agent 工作流。接著用學校試辦驗證管理端與系統整合。To C 則選少數高價值主題上線，不要一開始做成無邊界的大型學習平台。

0知識主題整理定義可學習主題、格式、metadata、權限與品質標準。
1Internal MVP課綱對照、題目標記、教材轉製，選一條流程先跑通。
2To B Pilot一校／一班／一門課，完成教師端、行政端、資料整合。
3To C 主題試營運選 3–5 個主題，驗證留存、完成率、回訪與付費。
4平台化累積可重複使用的 Engine、Connector、內容格式與 Agent Template。

每一階段的決策重點

階段要回答的問題產出物不該做的事
知識主題整理 哪些主題值得變成 KALM 的第一批內容？內容結構如何標準化？ 主題清單、metadata、內容格式、品質標準。 不要先把所有教材都丟進 RAG，然後期待自動變產品。
Internal MVP Agent 是否真的能降低企劃／編輯工作成本？ 一條可展示工作流、前後效率比較、人工修正紀錄。 不要只做聊天介面，沒有接上真實文件與流程。
To B Pilot 學校是否能導入、管理、追蹤、匯出報表？ 學生端、教師端、行政端、整合規格、驗收報告。 不要只給學生試用，卻沒有老師與行政可管理的工具。
To C 試營運 使用者是否願意持續學、回來學、甚至付費？ 主題頁、學習流程、練習、回饋、個人學習紀錄。 不要做成開放式 AI 問答站，與大型模型平台正面衝突。
給會議用的收斂問題

建議把第三點討論收斂成 6 個決策題

這些問題不是發散用，而是用來逼出下一步分工、MVP 與資源配置。

1 第一批知識主題要選哪些？

建議不要超過 3–5 個。可從課本內容、商業口說英文、買車買房、保險理財等方向挑選。

2 To C 是否先做「封閉主題式學習」？

也就是使用者從既有知識主題進入，而不是完全開放式任意問答。

3 To B 第一個學校場景是什麼？

一校、一班、一門課、一組主題。需要定義老師端、行政端與驗收報表。

4 校務系統整合先採哪種路徑？

API、Raw Data 匯入、批次同步、SSO，哪一個最符合第一個試辦場景？

5 Internal MVP 先挑哪條工作流？

課綱對照、題目標記、教材轉製、版本比對，先選一條可以量化效率提升的流程。

6 Personal Device 先支援哪種形態？

不要先糾結 AI PC。應先確認場景需要的是桌面、手機、平板、眼鏡，還是混合式體驗。


可直接在會議中使用的發言版本

我建議第三點的 PMF 不要先從硬體或模型切入。硬體、模型、RAG、MCP 都是支撐能力，不是市場本身。

KALM 比較適合先收斂成三個場景：
To C 是基於已整理知識主題的個人 Learning Agent
To B 是可管理、可整合校務系統、可驗收的學校導入方案
Internal 則是出版社內部教材生產與知識管理 Agent。

短期建議先用內部真實教材與工作流驗證，再做學校試辦，最後挑少數高價值主題做 To C 試營運。
這樣討論會比較能收斂到內容資產、系統整合、驗收指標與分工，而不是一直繞在裝置或模型規格上。
一句話總結

Agent KALM 的 PMF，應該是
「可信任知識主題 × 學習流程 × 管理整合 × 個人裝置」

To C 要讓個人沿著整理好的主題真的學會
To B 要讓學校可以管理、整合、驗收
Internal 要讓出版社把教材資產轉成可持續生產、可外部商業化的知識服務。
