產品企劃發想｜影片課程 AI 完課增強方案

HaHow 影片課程 AI 完課增強方案

以 HaHow 既有課程內容、會員入口與觀看行為為基礎，透過外部 LLM／RAG 能力建立低成本、限量使用、以影片為核心的 AI 學習輔助模組。 目標不是取代課程，而是提升看課回訪、章節理解、完課率與課後吸收感。

版本：企劃發想 v4 日期：2026/05/28 適用：合作提案 / PoC 規劃 / 內部討論 產品定位：輕量 AI 增強層

一、企劃摘要

核心目標

讓影片課更容易被看完

以章節重點卡、小測驗、站內／LINE 提醒與有限問答，降低學員中斷率，讓使用者從「買課」更容易走到「持續學習」。

成本策略

AI 只在關鍵處出手

大量內容採批次預生成，使用者端即時 LLM 僅用於限量答疑與少量任務簡評，避免無限制聊天造成成本黑洞。

合作模式

我方主導產品與整合

我方負責產品規劃、資料需求定義、API contract、三方協調、驗收規格與商業模式設計；LLM 廠商提供模型與 RAG 能力。

一句話定位：影片是主菜，AI 是導覽、提醒、整理、有限解惑與學習確認工具。

二、合作架構與權責界面

本方案適合採三方協作：HaHow 提供課程與使用場景，LLM 外包商提供模型與知識檢索能力，我方負責產品方案、資料規格、流程整合與驗收管理。

五、產品模組設計

模組 A｜低 AI 成本

AI 章節重點卡

每個章節提供「本章重點、常見誤解、看完應會、複習段落」。由 LLM 批次生成後進入審核流程。

  • 適合全課程導入
  • 不消耗即時問答成本
  • 可放在播放器旁或章節頁
模組 B｜低 AI 成本

AI 生成章節小測驗

每章 3～5 題選擇題／判斷題／情境題。作答與批改採規則式，AI 僅用於題目生成與答案解析。

  • 用來確認理解
  • 可提升章節完成感
  • 適合接入完課率分析
模組 C｜低營運成本

LINE / 站內完課提醒

依照觀看進度與中斷天數推送提醒、複習卡、小測驗或回課程連結。文案可預生成與模板化。

  • 主攻回訪率
  • 不需要即時 LLM
  • 可配合 HaHow 既有 LINE 綁定
模組 D｜限量 AI

課程限定 AI 問答

每門課提供免費 AI 點數，付費加購採點數制而非固定題數。一般短問答消耗 1 點，長篇解析、總複習與任務簡評依成本消耗較多點數；回答必須引用課程章節，並導回影片。

  • 適合高單價與高卡關課程
  • 採點數制控管成本
  • 可建立常見問題快取
模組 E｜第二階段

課後任務與 AI 簡評

每門課設計 1～3 個文字型任務，由 AI 依課程框架給簡短回饋。不做重度批改，不處理複雜檔案。

  • 從「看完」推向「做出來」
  • 每人限 1～3 次簡評
  • 適合職場、英文、AI 工具課
模組 F｜企業加值

輕量學習成效報告

提供企業 HR 觀看完成、測驗通過、常見卡點與匿名化學習摘要。不取代完整 LMS，只作為銷售與續約輔助。

  • 適合 HaHow for Business
  • 每週／每月批次生成
  • 避免即時大報告成本

六、資料與 API 需求

PoC 成敗高度取決於資料取得與 API 串接成熟度。建議在開發前先完成 API contract 與資料授權確認。

HaHow 端建議提供資料 / API
  • 課程資料：course_id、課程名稱、講師、分類、售價、課程狀態。
  • 章節資料：lesson_id、章節標題、順序、影片長度、時間軸。
  • 教材資料：逐字稿、講義、投影片、作業說明、FAQ。
  • 會員權限：匿名 user_id、是否購買、是否可使用 AI 模組。
  • 觀看進度：開始觀看、觀看比例、最後觀看時間、完課狀態。
  • 入口整合：播放器旁 widget、章節頁嵌入、deep link 回課程。
  • 通知能力：LINE 綁定狀態、站內通知、App 推播或 email 事件。
LLM 外包商建議提供 API
  • 資料匯入：課程內容 ingestion、chunking、向量化、metadata 設定。
  • 批次生成：章節摘要、小測驗、答案解析、FAQ、提醒文案。
  • 課程問答：限定課程範圍回答、引用來源、回答字數限制。
  • 簡評：依任務 rubric 產生短回饋與下一步建議。
  • 用量紀錄：token、請求次數、模型版本、成本估算、錯誤碼。
  • 品質回報：低信心回答、無資料回答、使用者負評回收。
資料類型 用途 必要性 風險 / 注意事項
逐字稿 摘要、測驗、問答、引用來源 高 若無逐字稿，需要先轉寫或改用講義與章節摘要，品質會下降。
課程章節與時間碼 導回影片、產生複習段落 高 若時間碼不足，AI 無法精準導回播放器位置。
觀看進度 提醒、完課分析、A/B test 高 需確認事件定義一致，例如何謂完成章節、何謂回訪。
購課權限 限制只有購買者可使用 AI 高 需避免未購買者透過 AI 取得完整課程內容。
LINE / 通知狀態 召回與提醒 中 若短期無法整合，可先用 email 或站內提醒替代。
學生提問紀錄 FAQ 快取、卡點分析 中 需匿名化與設定資料保留期間。

七、AI 使用量與成本控管

批次型 AI

用於課程上架前或定期更新。

  • 章節摘要
  • 小測驗
  • FAQ 初稿
  • 提醒文案

成本：低，可預估，可攤提。

即時型 AI

用於使用者提出個人問題。

  • 課程限定問答
  • 少量追問
  • 任務簡評

成本：中，需要額度、快取與回答長度限制。

高成本 AI

第一階段不建議開放。

  • 無限聊天
  • 長篇報告
  • 圖片／影片批改
  • 程式執行與除錯環境

成本：高，應放到付費方案或後期版本。

控管機制 設計方式 目的
點數制 每門課提供免費 3～5 點；加購採 AI 點數包。一般問答 1 點，長篇解析 2～5 點，任務簡評 15～25 點；企業版依合約配額。 避免無限制問答。
語意快取 相似問題優先回 FAQ / cache，不重複呼叫 LLM。 降低熱門課程反覆問答成本。
回答長度限制 預設 300～800 字，不提供整門課完整代讀。 控 token，也避免取代課程。
課程範圍限制 只回答該使用者已購買課程內容，超出範圍則引導縮小問題。 保護課程價值與授權邊界。
導回影片 回答結尾附對應章節與時間段。 讓 AI 成為影片入口，而非影片替代品。
使用量儀表 依課程、使用者、功能、模型記錄成本。 便於報價、分潤與成本預警。

八、MVP 建議範圍

第一階段不建議做完整 App、完整 LMS、無限 AI 家教或企業大後台。建議先用 3 門課驗證完課增強效果。

MVP 建議包含

  • 3 門課程資料匯入與 AI 處理
  • 章節重點卡
  • 章節小測驗
  • 站內或 LINE 提醒機制擇一
  • 每位使用者每門課免費 3～5 點 AI 額度
  • AI 點數額度、快取與使用量紀錄
  • 基礎成效報表：觀看、回訪、測驗、問答使用

MVP 不建議包含

  • 無限制 AI 聊天
  • 完整企業 LMS 後台
  • 圖片、影片、聲音作業批改
  • 程式碼執行環境
  • 多角色複雜權限管理
  • 大規模全站課程自動導入
  • 深度個人化學習路徑重算

九、開發週期規劃

以下以「三方協作、3 門課 PoC、需串接 HaHow API 與 LLM 廠商 API」為估算前提。實際時程取決於 HaHow API 成熟度、課程資料完整度與 LLM 廠商可用能力。

Phase 0｜需求確認與合作規格 2～3 週

確認 PoC 課程、使用者範圍、資料授權、功能邊界、成功指標、三方窗口與驗收方式。

Phase 1｜API Contract 與資料沙盒 3～5 週

定義 HaHow → 我方、我方 → LLM 廠商的 API 規格，建立測試資料、權限驗證、log 與錯誤處理流程。

Phase 2｜課程內容處理與預生成 4～6 週

處理逐字稿、章節 metadata、講義，建立 RAG 知識庫，產出章節重點卡、小測驗、答案解析與 FAQ 初稿。

Phase 3｜前台模組與提醒機制 4～6 週

建置播放器旁或課程頁元件、測驗流程、問答入口、站內／LINE 提醒、deep link 回課程。

Phase 4｜額度、快取與成本儀表 3～5 週

建立問答配額、cache、使用量紀錄、token / 請求成本回收、基本營運報表。

Phase 5｜UAT、A/B Test 與上線 3～4 週

三方測試、資料正確性、AI 回答品質、權限與成本壓測、小規模使用者導入與成效追蹤。

建議 PoC 總週期：12～18 週。若 HaHow API 與資料格式成熟，可壓至 10～14 週；若需大量平台客製，可能延長至 20 週以上。

企業版月費 企業客戶加購 AI 成效模組 客單價高，續約價值大。 企業權限與資料隱私要求較高。
每門課 AI 化費用 依課程收取資料處理與設定費 可反映內容處理成本。 需建立標準化工具，否則人力成本高。

十三、主要風險與處理策略

風險 影響 處理策略
HaHow API 或資料不足 無法精準導回影片、提醒與權限控管。 先用匯出資料與 deep link 做 PoC；正式版再深化 API。
逐字稿品質不足 摘要、測驗與問答品質下降。 建立逐字稿清洗流程與人工抽審機制。
LLM 成本失控 加購收入無法覆蓋營運成本。 點數制、快取、短答、批次生成、成本儀表。
AI 回答取代課程 傷害影片觀看與老師內容價值。 回答限制字數，並固定導回章節與時間碼。
課程授權不清 講師或平台內容使用爭議。 上線前確認資料可用範圍、保存期限、生成內容權利歸屬。
三方溝通成本高 時程延誤與責任不清。 建立單一 issue tracker、API 文件、驗收表、週會節奏與問題分級。

十四、建議下一步

第一步：先做合作需求盤點

  • 確認 HaHow 願意提供哪些資料與 API。
  • 確認 3 門 PoC 課程與講師授權。
  • 確認使用者入口：播放器旁、章節頁、站內通知或 LINE。
  • 確認 PoC 成功指標。

第二步：要求 LLM 廠商提供技術能力清單

  • 支援哪些資料格式與 ingestion 流程。
  • 是否可回傳引用來源與信心分數。
  • 是否提供 token / request 成本報表。
  • 是否支援快取、模型切換與回答長度限制。
對外提案主軸：用低成本 AI 增強 HaHow 既有影片課程，先用 3 門課驗證回訪率、章節完成率、測驗完成率與限量問答的商業可行性。
